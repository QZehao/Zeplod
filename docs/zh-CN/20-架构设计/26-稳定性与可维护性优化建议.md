# Zeplod 稳定性与可维护性优化建议

本文基于当前主项目代码（不含 `src/proprietary`）的静态审查，目标是给出可落地的重构与优化路线，优先提升稳定性，其次提升可维护性与演进效率。

## 1. 结论摘要

当前系统在功能完整性上较高，但在以下方面已经出现“规模化维护”信号：

- 核心模块文件体积过大，职责聚合过多，修改风险高。
- 生命周期与并发控制逻辑在多个模块重复实现，行为一致性依赖人工保证。
- 个别路径存在轮询等待（busy-wait）或超时后降级处理，易引入边界不稳定。
- 测试中存在非确定性断言与时间敏感写法，影响回归检测可靠性。

建议按 `P0 -> P1 -> P2` 分阶段推进，先夯实测试，再拆分核心模块。

## 3. 分级优化建议

## P0（立即执行，优先稳态）

1. 先修测试，再改实现
- 清理永真断言与“实现定义”断言，改为明确契约。
- 把关键用例从 `k_msleep` 驱动改为信号量/事件驱动。
- 用真实 stub callback 替代伪函数地址。

2. 建立统一异常行为契约
- 明确 `NULL callback`、`cancel race`、`stop during pending` 等边界返回值。
- 在头文件注释与测试同步固化。

## P1（核心重构，降低维护风险）

1. 拆分超大文件（先机械拆分，不改语义）
- `event_system.c` 建议拆为：
  - `event_system_lifecycle.c`
  - `event_system_pubsub.c`
  - `event_system_memory.c`
  - `event_system_stats.c`
- `module_manager.c` 建议拆为：
  - `module_manager_lifecycle.c`
  - `module_manager_dependency.c`
  - `module_manager_event_binding.c`
  - `module_manager_stats.c`
- `ipc_service.c` 建议拆为：
  - `ipc_service_lifecycle.c`
  - `ipc_service_call.c`
  - `ipc_service_future.c`
  - `ipc_service_pending.c`

2. 合并重复路径
- 抽象 `event_create_with_data*` 与 `event_publish_copy*` 的共用内部函数，降低双路径漂移风险。

## P2（架构收敛与性能稳定）

1. 抽取统一 lifecycle runner
- 统一线程服务模块的 `init/start/stop/deinit + join/abort/timeout` 模式。
- 统一日志与错误上报，减少模块间行为差异。

2. 替换忙等为事件驱动等待
- 优先处理 `event_system` 与 `data_bus` 的轮询等待点。
- 目标是降低 CPU 抖动并提升边界可预测性。

3. 错误码域统一
- 建议逐步向 errno 风格收敛（或建立统一 domain code），降低跨模块映射复杂度。

## 4. 推荐 PR 切分（低风险推进）

1. `test: stabilize flaky and ambiguous assertions`
- 只改测试，不改实现。

2. `refactor(event): split event_system by responsibility`
- 仅搬移代码与补 include，不改行为。

3. `refactor(module): extract dependency planner`
- 把拓扑/依赖解析从 `module_manager.c` 独立。

4. `refactor(ipc): split lifecycle and call paths`
- 清晰分离 stop/cancel/future 流程。

5. `refactor(core): unify lifecycle runner for threaded services`
- 让 dispatcher、ipc、data_bus 共享模板化生命周期骨架。

6. `perf(core): remove busy-wait loops with event-driven waits`
- 逐点替换，保留回归测试。

## 5. 验收标准（建议）

- 全量 `tests/` 在 `native_sim` 与 CI 对齐板型均稳定通过。
- 长时间循环（1000 次 start/stop）无卡死、无 join 超时。
- 关键模块文件行数显著下降，单文件职责边界清晰。
- 关键行为（cancel/timeout/stop race）有明确且稳定的测试断言。

---

维护建议：后续每次新增并发路径时，同步补充“生命周期状态图 + 锁序说明 + 对应用例”，避免复杂度再次回流到单文件。
