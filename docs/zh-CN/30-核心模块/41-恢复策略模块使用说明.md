# 恢复策略模块使用说明（recovery_policy）

本文说明可选 **`recovery_policy`** 模块（`CONFIG_RECOVERY_POLICY_MODULE`）：监听 `module_manager` 的 `MODULE_MGR_EVENT_ERROR`，对失败模块执行分级恢复（单模块重启、全量重启或暖复位）。

**相关**：[32-模块系统详细使用说明.md](32-模块系统详细使用说明.md) · [Phase 2 计划](../../superpowers/plan/2026-06-13-phase2-recovery-secure-mcuboot-plan.md)

---

## 1. 启用

```bash
west build -b <board> . -- -DEXTRA_CONF_FILE=conf/features/recovery.conf
```

依赖 `CONFIG_MODULE_MANAGER=y`。自动注册由 `CONFIG_RECOVERY_POLICY_AUTOINIT` 控制（默认 `y`）；单元测试请设 `n` 并手动 `recovery_policy_init()` / `start()`。

---

## 2. 恢复策略

| Kconfig | 行为 |
|---------|------|
| `RECOVERY_POLICY_ACTION_RESTART_MODULE`（默认） | `stop` → `clear_error` → `start` 失败模块 |
| `RECOVERY_POLICY_ACTION_RESTART_ALL` | 超过单模块重试上限后 `clear_all_error` → `stop_all` + `start_all` |
| `RECOVERY_POLICY_ACTION_REBOOT` | 超过上限后 `sys_reboot(WARM)` |
| `RECOVERY_POLICY_ACTION_NONE` | 仅日志 |
| `RECOVERY_POLICY_OTA_HOOK`（默认 y，需 OTA） | 订阅 `EVENT_OTA_STATE_CHANGED`，OTA 进入 ERROR 时恢复 |
| `RECOVERY_POLICY_WATCHDOG_HOOK`（默认 y，需 WDT） | 看门狗过期警告前尝试 `restart_all` |
| `SYS_FAULT_DUMP_ENABLE`（recovery 默认 y） | `.noinit` RAM 记录最近故障事件 |

单模块自动重启次数由 `CONFIG_RECOVERY_POLICY_MAX_RESTARTS`（默认 3）限制；未超限时始终先尝试 **单模块重启**。

`restart_all` 会先调用 `module_manager_clear_all_error_states()`，确保 ERROR 模块可被 `start_all` 再次拉起。

---

## 3. 与 module_manager 协作

恢复前会调用 `module_manager_clear_error_state()`，将 `MODULE_STATUS_ERROR` 置为 `STOPPED`，以便再次 `start_module`。

模块管理器全局回调仅支持一个消费者；`recovery_policy_start()` 会注册自身回调，`shutdown()` 时清除。

---

## 4. API

```c
int recovery_policy_on_module_error(uint32_t module_id);
uint32_t recovery_policy_get_restart_count(uint32_t module_id);
void recovery_policy_reset_restart_counts(void);
int recovery_policy_control(int cmd, void* arg);  /* RECOVERY_POLICY_CMD_RESET_COUNTS */
```

`module_manager_clear_all_error_states()` 供 `restart_all` 与测试使用。

`recovery_policy_on_module_error` 与 ERROR 事件回调路径相同，供 ztest 直接调用。
