## 1. 最终状态

本计划的终点不是“把所有代码重写一遍”，而是让 Zeplod 达到下面的工程状态：

1. 核心模块职责清晰  
   `event_system`、`module_manager`、`ipc_service` 不再是单个超大文件，而是按生命周期、订阅/发布、内存、统计、依赖规划、pending/future 等职责拆分。单文件改动不再牵动整个核心模块。

2. 并发生命周期行为统一  
   所有有工作线程的服务都遵循一致的 `init -> start -> stop -> deinit/shutdown` 模式，包括 stop 唤醒、join 超时、retry join、abort 策略、资源 drain、错误日志和可恢复边界。

3. 热路径性能边界明确  
   事件发布、事件分发、Data Bus 发布/分发、IPC pending 查找等路径的锁、扫描、内存分配和等待策略都有明确约束。常用路径避免不必要的全表扫描和轮询等待。

4. 测试可作为重构护栏  
   并发、超时、cancel、stop during pending、生命周期循环等行为有确定性测试覆盖。测试尽量使用信号量、事件或显式状态同步，减少 `k_msleep()` 驱动的偶然通过。

5. 扩展成本可控  
   新增模块、新增事件类型、新增 Data Bus channel、新增 IPC 调用模式时，有清楚的接入点、错误码契约、锁序约定和测试模板。

## 2. 非目标

本计划刻意不包含以下事项：

- 不改变公开 API 的语义，除非现有语义已经被测试证明存在 bug。
- 不一次性替换所有错误码体系；先建立边界映射，再逐步收敛。
- 不把所有静态数组改成动态分配；Zephyr/嵌入式场景下，静态容量仍然是可接受的设计。
- 不在同一个 PR 中同时做大规模文件拆分和行为变更。
- 不纳入 `src/proprietary/`，除非后续明确要求。

## 3. 当前主要问题

### 3.1 复杂度集中

当前最重的几个文件：

| 文件 | 问题 |
| --- | --- |
| `src/core/event_system.c` | 生命周期、发布订阅、事件内存、统计、兼容层辅助逻辑混在一起 |
| `src/modules/module_manager.c` | 模块注册、生命周期、依赖排序、事件绑定、统计调试集中在一个文件 |
| `src/modules/ipc_service/ipc_service*.c` | 已拆：lifecycle / worker / dispatcher / pending / call + `internal.h` |
| `src/core/event_queue.c` | 队列控制块、溢出策略、统计、deinit/ISR 约束混合 |

### 3.2 重复的线程生命周期模式

以下模块都在各自实现线程启动、停止、唤醒、join、超时处理：

- `event_dispatcher`
- `ipc_service`
- `data_bus`

这些代码的语义相近，但细节不完全一致，长期会带来边界行为漂移。

### 3.3 热路径仍有可优化点

- `event_dispatcher` 主循环多次拿锁读取 state/config。
- `data_bus` dispatcher 每次被唤醒后扫描所有 channel。
- `ipc_service` pending 表按 request id 线性扫描。
- `module_manager` 多处按 `CONFIG_MAX_MODULES` 建栈上快照。
- 部分等待路径使用 sleep/poll，边界延迟不够精确。

### 3.4 测试确定性不足

部分测试依赖 `k_msleep()`，并发结果允许“0 或 1”这类宽松断言。它们适合烟测，不适合作为大规模重构的回归护栏。

---

## 4. 推进原则

1. 先测试，后重构  
   对生命周期和并发路径，先补确定性测试，再做行为变更。

2. 先机械拆分，后抽象  
   大文件先按职责搬迁函数，不改行为；等边界稳定后再合并重复逻辑。

3. 每个 PR 只改变一个维度  
   例如“只拆文件”、“只改 pending 表”、“只改 Data Bus ready queue”，避免回归定位困难。

4. 保留嵌入式可预测性  
   优先使用静态容量、显式状态、有限等待和编译期约束。动态分配只用于已有设计允许的路径。

5. 对外兼容优先  
   新 API 可补充，旧 API 保留包装层；弃用需要文档和迁移说明。

## 5. 阶段计划

### P0：建立可靠护栏

目标：让后续重构有稳定回归基础。

任务：

1. 梳理异步测试等待方式  
   将关键 `k_msleep()` 改为 `k_sem`、`k_event` 或原子状态 + 有界等待 helper。

2. 清理模糊断言  
   将“可能成功也可能失败”的断言改成明确契约，或拆成两个可控场景测试。

3. 替换伪 callback 地址  
   用真实 stub callback 替代 `(event_callback_t)0x1000` 之类不可调用地址。

4. 增加生命周期循环测试  
   覆盖 `event_system`、`event_dispatcher`、`ipc_service`、`data_bus` 的多轮 start/stop/deinit。

5. 固化边界行为表  
   在头文件或测试 README 中明确 cancel、timeout、stop during pending、重复 stop 的返回值。

验收标准：

- `tests/` 在 Linux/WSL `native_sim` 或 CI 对齐板型稳定通过。
- 新增测试不依赖裸 `k_msleep()` 判断异步完成。
- 关键并发行为失败时能给出确定性断言。

建议 PR：

```text

```

### P2：统一生命周期与错误边界

目标：减少不同线程服务之间的行为漂移。

任务：

1. 设计 `zepl_thread_service` 内部模式  
   不一定先抽公共 C 文件，可以先形成内部约定：

   - `initialized`
   - `running`
   - `stopping/shutdown`
   - `thread_started`
   - `thread_gen`
   - wake signal
   - join timeout
   - retry join 策略
   - drain 策略

2. 对齐三个线程服务  
   逐步让 `event_dispatcher`、`ipc_service`、`data_bus` 的 stop/deinit 语义一致。

3. 统一日志与错误处理  
   对 join timeout、重复 stop、当前线程调用 stop、deinit 时仍 running 等场景定义一致策略。

4. 建立生命周期状态图  
   在文档中为每个线程服务画出合法状态迁移和非法调用返回值。

验收标准：

- 三个线程服务均支持重复 stop 的可预测行为。
- join timeout 后可以安全重试或明确进入 ERROR 状态。
- 不允许从本服务线程内部 stop/deinit 的行为一致。

建议 PR：

```text
refactor(core): standardize threaded service lifecycle
```

### P3：热路径性能优化

目标：减少不必要扫描、锁竞争和轮询等待。

#### P3.1 Data Bus ready queue

现状：dispatcher 被唤醒后快照并扫描所有 channel。

目标：publish 时将有数据的 channel 标记为 ready，dispatcher 只处理 ready channel。

设计要点：

- 每个 channel 增加 `ready` 原子标志，避免重复入 ready queue。
- 全局 ready ring 存 channel 指针或 index。
- dispatcher drain 完一个 channel 后，如果仍有数据，再次入队或继续处理。
- destroy/deinit 需要处理 ready queue 中悬挂 channel 的生命周期。

验收标准：

- channel 数较大但活跃 channel 少时，dispatcher 扫描成本下降。
- destroy 与 publish/dispatch 并发无 UAF。
- ISR publish 路径不持 mutex。

建议 PR：

```text
perf(data-bus): dispatch only ready channels
```

#### P3.2 IPC pending 索引

现状：pending find/cancel/response dispatch 多处线性扫描。

目标：封装 pending table，降低重复扫描与一致性风险。

可选方案：

1. 低风险方案：只抽 `ipc_pending_table_*` API，内部仍线性扫描。
2. 中风险方案：增加固定容量 hash/index，request_id 映射到 pending slot。
3. 高风险方案：pending slot + intrusive list/free list + generation token。

建议先做方案 1，再评估方案 2。

建议 PR：

```text
refactor(ipc): isolate pending table operations
```

#### P3.3 Dispatcher state 快路径

现状：dispatcher 主循环频繁用 mutex 读取 state/config。

目标：将只读或热读字段拆成 atomic/const 快路径。

边界：

- `state` 可以考虑 atomic。
- `config` init 后不变，可明确 const 语义。
- `filter/filter_user_data` 仍需一致性保护。
- `thread_started/thread_gen` 仍由锁保护。

建议 PR：

```text
perf(event): reduce dispatcher hot-path locking
```

## 8. 验收标准

### 8.1 功能验收

- 主应用可构建。
- `tests/` 在 CI 对齐环境中通过。
- 所有公开 API 的既有行为保持兼容，新增 API 有文档。
- autoinit 与手动 init 两种模式均可用。

### 8.2 稳定性验收

- 1000 次 `init/start/stop/deinit` 循环无死锁、无 join timeout、无资源泄漏。
- stop 与 publish/cancel/dispatch 并发时结果可预测。
- `CONFIG_EVENT_QUEUE_OVERFLOW_*` 三种策略均有覆盖。
- ISR 安全路径不调用 mutex，不做动态堆分配。

### 8.3 性能验收

- Data Bus dispatcher 不再每次唤醒扫描所有 channel。
- IPC response dispatch 对 pending 表的查找路径被封装，可进一步替换索引实现。
- Event dispatcher 主循环锁次数减少，且 filter 一致性不回退。
- 热路径新增日志受 `CONFIG_*_LOG_LEVEL` 控制，不引入默认噪声。

### 8.4 可维护性验收

- 核心大文件拆分后，每个文件职责单一。
- 内部 helper 命名表达锁前置条件，例如 `_locked`、`_isr`、`_snapshot`。
- 头文件只暴露必要 API，内部结构放入 `*_internal.h`。
- 新增模块有接入 checklist 和测试模板。

## 9. 风险与缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| 文件拆分引入链接错误 | static/extern 边界不清 | 先机械拆分，小步编译 |
| 生命周期统一改变行为 | stop/deinit 返回值变化 | 先写契约测试，再改实现 |
| Data Bus ready queue UAF | channel destroy 与 ready dispatch 并发 | 保留 dispatch_hold/publish_hold，新增 ready flag 测试 |
| IPC pending 索引不一致 | cancel/timeout/response race | 先封装 API，再替换实现 |
| 性能优化破坏可读性 | atomic 与 mutex 语义混用 | 在 internal header 写明字段所有权 |
| 测试矩阵变慢 | CI 时间上升 | 分 quick/full 两档运行 |

## 10. 新模块接入 Checklist

新增核心模块或线程服务时，应满足：

1. 明确生命周期状态图。
2. 明确哪些字段由 mutex 保护，哪些字段可 atomic 读取。
3. 明确 ISR 可调用 API 列表。
4. 明确 stop/deinit 是否允许从本模块线程调用。
5. 明确队列满、内存不足、重复调用的返回值。
6. 提供至少一个正常流程测试和一个 stop/cancel/timeout 边界测试。
7. 文档中列出 Kconfig 开关、默认值和资源成本。

## 11. 完成定义

当以下条件全部满足时，本路线图可视为完成：

- `event_system`、`module_manager`、`ipc_service` 已按职责拆分为多源文件（各保留薄 `*.c` 入口）。
- 线程服务生命周期行为有统一契约和测试覆盖。
- Data Bus 分发不再依赖全 channel 扫描。
- IPC pending/future 逻辑已独立封装。
- 关键异步测试不再依赖裸 sleep。
- 构建系统按模块组织，测试可按模块运行。
- 文档、Kconfig、测试和实现保持同步。
