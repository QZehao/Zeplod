## Overview

**Thread IPC Service** (repository path `src/modules/ipc_service/`) is a "request-response" style service framework implemented **within a single-core application process**:

- A **Worker thread** dequeues messages from the request queue, calls your registered **`ipc_service_func_t`** to process business, and puts results into the response queue.
- A **Dispatcher thread** dequeues results from the response queue, matches pending calls by `request_id`, and completes **synchronous wakeup**, **asynchronous callback**, or **Future semaphore release**.

Three client invocation modes supported:

| Mode | Behavior Summary |
|------|------------------|
| **Sync** | Block until result returns or timeout (`k_sem_take`) |
| **Async** | Return immediately, result called in Dispatcher thread |
| **Future** | Return `ipc_future_t *`, can `ipc_future_wait` or poll `ipc_future_is_ready` |

### Key Features

| Feature | Description |
|---------|-------------|
| No kernel heap dependency | Queue buffers and dual thread stacks are embedded in `ipc_service_t`, no `k_malloc` / `k_calloc` |
| Kconfig master switch | `CONFIG_THREAD_IPC_SERVICE` |
| Optional example | `CONFIG_THREAD_IPC_SERVICE_EXAMPLE` |
| Optional event bridge | `CONFIG_THREAD_IPC_SERVICE_EVENT_BRIDGE`: Publishes processing results to the project's Event System |
| Thread model | Worker + Dispatcher, both with same priority (configured by `CONFIG_THREAD_IPC_SERVICE_PRIORITY`) |

## Differences from Zephyr Upstream IPC Service

Zephyr has **`CONFIG_IPC_SERVICE`** under the **Inter Processor Communication** menu, used for **multi-core / RPMsg / icmsg etc. backends** IPC Service subsystem.

This module intentionally uses **different Kconfig symbols**:

- This module: `THREAD_IPC_SERVICE` → `CONFIG_THREAD_IPC_SERVICE`
- Upstream: `IPC_SERVICE` → `CONFIG_IPC_SERVICE`

**Do not mix these names**. In `prj.conf`, use `CONFIG_THREAD_IPC_SERVICE=y` to enable this module; under normal circumstances **should not** enable `CONFIG_IPC_SERVICE` for this module (unless you really need Zephyr's multi-core IPC Service).

Header file `ipc_service.h` triggers a compile-time `#error` when **`CONFIG_THREAD_IPC_SERVICE`** is not enabled, preventing accidental inclusion.

## Architecture and Data Flow

```
  Calling Thread                Worker Thread              Dispatcher Thread
      |                            |                            |
      | ipc_call_*               |                            |
      |-------------------------->|  k_msgq(request_queue)     |
      |  Register pending + post  |  service_func()            |
      |                            |--------------------------->|
      |                            |  k_msgq(response_queue)   |  Match request_id
      |<---------------------------|                            |  Sync: k_sem_give
      |  Sync: sem wakeup          |                            |  Async: callback()
      |  Async: callback in Dispatcher |                            |  Future: future->sem
```

Key points:

- **Request ID** generated by `ipc_generate_request_id()`, **0 is reserved** (internally never allocates 0).
- **Async callbacks** execute in **Dispatcher context**, and **after releasing `pending_lock`**, avoiding deadlock with sync path.
- **Stopping service** posts a "dummy" message to each queue; both threads exit after checking `shutdown` upon dequeue, then `ipc_service_stop()` executes `k_thread_join` on both threads.

## Integration with Event System

This project **can** combine Thread IPC Service with **`event_system` (publish-subscribe)** in business logic. Typical use: After IPC processes requests in Worker, broadcast **result metadata** to multiple subscribed modules, not just the single `ipc_call_*` caller.

### Relationship Description

| Mechanism | Role |
|-----------|------|
| Thread IPC | Point-to-point request-response (with `request_id`, sync/async/Future) |
| Event System | One-to-many notification (subscribers each receive callback in event dispatcher thread) |

The two complement each other: IPC is suitable for **imperative RPC**; Events are suitable for **state/result broadcast** and **decoupling modules**.

### Method 1: Optional Event Bridge (Recommended, Implemented)

Add in `prj.conf`:

```conf
CONFIG_EVENT_SYSTEM=y
CONFIG_THREAD_IPC_SERVICE=y
CONFIG_THREAD_IPC_SERVICE_EVENT_BRIDGE=y
```

After **`event_system_init()`** and before publishing any events, call once (idempotent type registration):

```c
#include "ipc_service_event.h"

thread_ipc_event_register_types();
```

After **`event_system_start()`**, publish in `ipc_service_func_t`:

```c
static int my_ipc_service_func(ipc_request_id_t request_id,
			       const void *data, size_t data_size,
			       void **out_data, size_t *out_data_size)
{
	int ret = /* ... business processing ... */;

	(void)thread_ipc_event_publish_result(
		MY_SOURCE_ID,   /* e.g., module ID, for subscriber filtering */
		request_id,
		ret,
		EVENT_PRIORITY_NORMAL);

	*out_data = /* ... */;
	*out_data_size = /* ... */;
	return ret;
}
```

Subscribers use existing API:

```c
#include "event_system.h"

static void on_thread_ipc_result(const event_t *ev, void *user_data)
{
	const thread_ipc_event_result_t *p =
		(const thread_ipc_event_result_t *)ev->data;

	(void)user_data;
	/* Process based on p->source_id / p->request_id / p->result */
}

/* During initialization: event_subscribe(EVENT_TYPE_THREAD_IPC_RESPONSE, on_thread_ipc_result, NULL, &sub_id); */
```

Event type constant is defined in **`event_system.h`** as `EVENT_TYPE_THREAD_IPC_RESPONSE` (value `20`), payload structure is **`thread_ipc_event_result_t`** (defined in `ipc_service_event.h`).

**Note**:

- Bridge function only carries **fixed small structure** (`source_id`, `request_id`, `result`). If you need to attach large data blocks, call `event_publish_copy` / `event_create_with_data` with a **separate event type** inside `service_func`, and ensure `data` lifecycle until dispatcher thread consumption.
- Event System internally uses heap allocation for event bodies (not contradictory to "IPC module itself has no heap").

### Method 2: Without Enabling Kconfig, Directly Send Events in service_func

If you don't want to add `ipc_service_event.c`, you can directly `#include "event_system.h"` and call `event_publish_copy` etc. in `ipc_service_func_t`. Same requirement: publish **after `event_system_start()`**.

## Integration with Module Manager

**You can** register a "module holding an `ipc_service_t`" into **`module_manager`**, driving IPC Lifecycle in virtual table `init` / `start` / `stop`, with event subscription completed in `init`.

### Recommended Order (Illustration)

```
event_system_init()
  → thread_ipc_event_register_types()   (if EVENT_BRIDGE enabled)
  → module_manager_init() / each module init (ipc_service_init)
event_system_start()
  → module_manager_start()（ipc_service_start)
  → … run business …
module_manager_stop()（ipc_service_stop）
```

### `module_interface_t` and IPC Correspondence (Conceptual)

| Module Callback | IPC Operation |
|----------------|---------------|
| `init` | `ipc_service_init()`; can `event_subscribe` here |
| `start` | `ipc_service_start()` |
| `stop` | `ipc_service_stop()` |
| Before restart | Need `ipc_service_init()` again (see "Re-initialization" above) |

For specific field names, refer to `module_interface_t` in **`module_base.h`**; this repository's `example_module_a/b` can serve as reference for registration method, just replace business with `ipc_service_*`.

## Core Types

### Service Handler Function

```c
typedef int (*ipc_service_func_t)(ipc_request_id_t request_id,
                                  const void *data,
                                  size_t data_size,
                                  void **out_data,
                                  size_t *out_data_size);
```

- Called in **Worker thread**.
- Return value: success recommended `0`, failure is **negative errno style** (e.g., `-EINVAL`); this value is passed to client (Sync return value, Async callback `result`, Future `future->result`).
- `*out_data` / `*out_data_size`: Output buffer pointer and length; if no output needed, set `*out_data` to `NULL` and `*out_data_size` to `0`.

### Async Callback

```c
typedef void (*ipc_async_callback_t)(ipc_request_id_t request_id,
                                     int result,
                                     const void *data,
                                     size_t data_size,
                                     void *user_data);
```

Called in **Dispatcher thread**, **not** while holding `pending_lock`.

### Service Instance `ipc_service_t`

Contains: `k_msgq`, embedded queue buffer, **`worker_stack`** and **`dispatcher_stack`** defined by `K_KERNEL_STACK_MEMBER`, `pending_requests[]`, `futures[]`, mutex, etc. Instance size is large, suitable as **static global** or **long-lived** context member; note RAM usage (especially `THREAD_IPC_SERVICE_STACK_SIZE` doubled).

## API Reference

### Lifecycle

| Function | Description |
|----------|-------------|
| `int ipc_service_init(ipc_service_t *service, const char *name, ipc_service_func_t service_func, int priority)` | Initialize struct, message queues, mutex, Future free list. Stack and queue sizes determined by Kconfig: `CONFIG_THREAD_IPC_SERVICE_STACK_SIZE`, `CONFIG_THREAD_IPC_SERVICE_REQUEST_QUEUE_SIZE`, `CONFIG_THREAD_IPC_SERVICE_RESPONSE_QUEUE_SIZE`. |
| `int ipc_service_start(ipc_service_t *service)` | Create and start Worker and Dispatcher threads. Returns `-EALREADY` if already running. |
| `int ipc_service_stop(ipc_service_t *service)` | Set `shutdown`, wake both threads, `k_thread_join`; failure returns `-EIO` (join failed) or `-EINVAL` (NULL pointer). **Does not** release embedded buffers (follows instance lifecycle). Returns `0` directly if not started. |

### Invocation Interfaces

| Function | Description |
|----------|-------------|
| `int ipc_call_sync(..., k_timeout_t timeout)` | Sync call; `data` cannot be `NULL` (even if `data_size` is 0); `out_data` / `out_data_size` cannot be `NULL`. `K_NO_WAIT` / zero timeout returns `-EINVAL`. |
| `int ipc_call_async(..., ipc_async_callback_t callback, void *user_data, ipc_request_id_t *out_request_id)` | Async call; `callback` cannot be `NULL`; `out_request_id` can be `NULL`. |
| `int ipc_call_future(..., ipc_future_t **out_future)` | Allocate Future and post request; must `ipc_future_release` after success. |

### Future

| Function | Description |
|----------|-------------|
| `int ipc_future_wait(ipc_future_t *future, int *out_result, const void **out_data, size_t *out_data_size, k_timeout_t timeout)` | Wait for completion semaphore; output parameters can be `NULL` (as needed). |
| `bool ipc_future_is_ready(ipc_future_t *future)` | Non-blocking query of `completed` flag. |
| `int ipc_future_release(ipc_service_t *service, ipc_future_t *future)` | Return Future to pool; validates pointer belongs to that `service`'s `futures[]`, otherwise `-EINVAL`; returns `-EBUSY` if held by other code; returns `-EALREADY` if already released. |

### Utilities

| Function | Description |
|----------|-------------|
| `size_t ipc_service_get_pending_count(ipc_service_t *service)` | Current `in_use` pending entry count (locked statistics). |
| `int ipc_service_cancel(ipc_service_t *service, ipc_request_id_t request_id)` | Cancel request still in pending table; returns `-ENOENT` if not found. |
| `ipc_request_id_t ipc_generate_request_id(void)` | Global incrementing ID (skips 0); shared counter across multiple instances. |

### Event Bridge (`CONFIG_THREAD_IPC_SERVICE_EVENT_BRIDGE`)

| Function | Description |
|----------|-------------|
| `event_status_t thread_ipc_event_register_types(void)` | Register `EVENT_TYPE_THREAD_IPC_RESPONSE` (idempotent). Must be called after `event_system_init()`. |
| `event_status_t thread_ipc_event_publish_result(uint32_t source_id, ipc_request_id_t request_id, int result, event_priority_t priority)` | `event_publish_copy` with fixed payload; must be called after `event_system_start()`, in `service_func` and other thread contexts. |

## Shared Memory Invocation

After enabling `CONFIG_THREAD_IPC_SERVICE_SHARED_MEM`, provides a reference-counted Shared Memory pool, solving data ownership issues between caller and service function.

### Configuration Options

| Config Item | Type | Default | Description |
|-------------|------|---------|-------------|
| `CONFIG_THREAD_IPC_SERVICE_SHARED_MEM` | bool | n | Enable shared memory subsystem |
| `CONFIG_THREAD_IPC_SERVICE_SHARED_MEM_POOL_SIZE` | int | 8 | Memory block pool size |
| `CONFIG_THREAD_IPC_SERVICE_SHARED_MEM_BLOCK_SIZE` | int | 256 | Single block size (bytes) |
| `CONFIG_THREAD_IPC_SERVICE_SHARED_MEM_LOG_LEVEL` | int | 3 | Log level |

### Type Definitions

```c
typedef uint32_t ipc_shm_handle_t;  // Opaque handle
#define IPC_SHM_HANDLE_INVALID ((ipc_shm_handle_t)0xFFFFFFFFU)

typedef enum {
    IPC_SHM_STATE_FREE = 0,      // Free
    IPC_SHM_STATE_ALLOCATED = 1, // Allocated (refcount=1)
    IPC_SHM_STATE_SHARED = 2,    // Shared (refcount>1)
    IPC_SHM_STATE_INVALID = 3    // Invalid state
} ipc_shm_state_t;
```

### Shared Memory API

| Function | Description |
|----------|-------------|
| `ipc_shm_handle_t ipc_shm_alloc(ipc_service_t *service, size_t size)` | Allocate block, initial refcount=1; returns `IPC_SHM_HANDLE_INVALID` on failure |
| `void ipc_shm_acquire(ipc_service_t *service, ipc_shm_handle_t handle)` | Increment refcount (pass block to another consumer) |
| `int ipc_shm_release(ipc_service_t *service, ipc_shm_handle_t handle)` | Decrement refcount; auto-recycle when count=0; returns `-EINVAL` (invalid handle), `-EBUSY` (abnormal state) |
| `void *ipc_shm_get_ptr(ipc_service_t *service, ipc_shm_handle_t handle)` | Get block pointer; returns NULL for invalid handle |
| `size_t ipc_shm_get_size(ipc_service_t *service, ipc_shm_handle_t handle)` | Get block size |
| `bool ipc_shm_is_valid(ipc_service_t *service, ipc_shm_handle_t handle)` | Validate handle |
| `void ipc_shm_get_stats(ipc_service_t *service, uint32_t *active, uint32_t *peak, uint32_t *free)` | Get pool statistics |
| `int ipc_shm_init(ipc_service_t *service)` | Initialize shared memory pool |
| `void ipc_shm_deinit(ipc_service_t *service)` | Deinitialize, detect leaks |

### Usage Flow

```c
// Caller
ipc_shm_handle_t handle = ipc_shm_alloc(service, sizeof(my_data));
void *ptr = ipc_shm_get_ptr(service, handle);
memcpy(ptr, &my_data, sizeof(my_data));
// Pass handle via ipc_call_sync to service

// Service processes, returns handle via ipc_call_sync response

// Caller uses, then releases reference
ipc_shm_release(service, handle);
```

### Release Path Clarification

Only `ipc_call_sync_shm` returns handle to caller, and **`must`** call `ipc_shm_release()`. While caller holds handle, cannot call `ipc_shm_alloc` to get block of same instance again (will return new handle, original handle still valid).

## Three Invocation Modes

### 1. Sync (Synchronous)

Typical flow:

1. Call `ipc_call_sync`, internally register pending, post request, block on `response_sem`.
2. Worker processes, Dispatcher `k_sem_give` wakes calling thread.
3. Calling thread reads `out_data` / `out_data_size` and function return value (i.e., `service_func` result).

On timeout or failure, pending is released; if request was sent but timeout occurred, a "late response" may arrive finding no pending (warning log), which is expected behavior.

### 2. Async (Asynchronous)

- Call returns `0` immediately (only means **enqueue succeeded**, not business success).
- Business completion executes `callback` in **Dispatcher** thread.
- **Do not** block for long in callback; if calling same service's sync interface in callback, note priority and deadlock risks.

### 3. Future

- After `ipc_call_future` succeeds, you get `ipc_future_t *`, can first poll `ipc_future_is_ready`, then `ipc_future_wait` (or just wait).
- Must `ipc_future_release` after use, otherwise leaks Future slot.

---

## Service Lifecycle and Re-initialization

- **Startup order**: `ipc_service_init` → `ipc_service_start`.
- **Stop**: `ipc_service_stop` joins both threads and sets `running` to false; **does not** `memset` entire `ipc_service_t`.
- **Reuse**: If you need to restart the service, call **`ipc_service_init`** again (internally `memset` entire instance and rebuild queues and state), then `ipc_service_start`. Do not directly `start` a `stop`'ed instance without `init`.

---

## Memory, Concurrency and Security

1. **Input buffer lifecycle**: `data` received by `service_func` points to memory passed by caller, must remain valid from **enqueue to before Worker returns**; similarly, if `out_data` points to caller stack or temporary buffer, caller must ensure it **remains valid until reading results**.
2. **Trust boundary**: `service_func` and async callbacks both run in **kernel threads**, should be treated as same trust domain as application; do not execute dangerous operations driven by untrusted data.
3. **`ipc_service_cancel`**:
   - **Sync**: Sets `result = -ECANCELED` and `k_sem_give`, pending released by `ipc_call_sync` path.
   - **Future**: Sets future result and `k_sem_give`, and releases pending.
   - **Async**: Only releases pending; if response arrives later, may log "no pending" warning, response discarded.
4. **ISR**: Current API uses `k_mutex` and `k_sem_take`, **should not** directly call `ipc_call_sync` / `ipc_call_async` / `ipc_call_future` in interrupt context (unless specific API has explicit ISR-safe documentation for that Zephyr version; this module is designed for thread usage).

---

## Best Practices

- Set queue depth and `MAX_PENDING_REQUESTS` according to peak concurrency; when queue is full, `k_msgq_put(..., K_FOREVER)` blocks calling thread.
- **`service_func` in Worker should complete quickly**; heavy computations should be posted to work queue or dedicated thread, avoid blocking entire service chain.
- Log module name registered as **`thread_ipc_svc`**, can filter by module name in log filtering for debugging.
- RAM tight: reduce `THREAD_IPC_SERVICE_STACK_SIZE`, queue depth, and pending count; note `ipc_service_t` contains **two** same-sized stacks, total stack overhead is about `2 * STACK_SIZE` plus queues and pending structures.

## Troubleshooting

| Symptom | Possible Cause | Resolution |
|---------|----------------|------------|
| Compile error `ipc_service.h requires CONFIG_THREAD_IPC_SERVICE` | Included header without Kconfig | Add `CONFIG_THREAD_IPC_SERVICE=y` to `prj.conf` or remove include |
| `ipc_service_init` returns `-EINVAL` | `stack_size` / queue parameters inconsistent with Kconfig | Use same values as `prj.conf`, or modify Kconfig first then code |
| Sync call blocks forever | Not `start`'ed, Worker not running, or `service_func` deadlock | Confirm `ipc_service_start`; check Worker stack and function implementation |
| `ipc_call_async` no callback | Dispatcher not running or pending lost | Confirm service started; check for cancel / timeout releasing pending |
| `ipc_future_release` returns `-EINVAL` | `future` not allocated by that `service` | Check pointer source and multi-instance mixing |
| Link or config shows Zephyr `IPC_SERVICE` related backend | Accidentally enabled `CONFIG_IPC_SERVICE` | This module only uses `CONFIG_THREAD_IPC_SERVICE` |
| Link `undefined reference to k_malloc` | `CONFIG_HEAP_MEM_POOL_SIZE=0` | Set **`CONFIG_HEAP_MEM_POOL_SIZE`** in `prj.conf` (e.g., 8192); Event System needs kernel heap |
| `region RAM overflowed` | Static stack/queue too large or multiple demos enabled | Reduce `THREAD_IPC_SERVICE_*`, do not enable both **`EXAMPLE`** and **`EXAMPLE_MODULE_THREAD_IPC`**, or switch to board with larger SRAM |

## Example Code Description

File: `src/modules/ipc_service/ipc_service_example.c` (compiled with `CONFIG_THREAD_IPC_SERVICE_EXAMPLE=y`).

- In `SYS_INIT`, calls `ipc_service_init` + `ipc_service_start`, and creates a demo thread that runs Sync / Async / Future in sequence.
